# VibeTether

Run `vibetether context --boundary <boundary> --json` before reading VibeTether state. Treat the returned capsule as the only default control context.

Follow confirmed Truth, the current bounded slice, blockers, the selected Provider, and only fresh applicable Experience. Use `context read <handle>` for one needed source instead of scanning the project control assets.

Classify the request before acting:

- keep an explicitly read-only or clearly bounded low-risk task lightweight;
- investigate discoverable repository facts autonomously;
- when one unresolved user-owned decision remains, ask only that decision with a concrete recommendation and wait;
- route high-ambiguity or high-impact work to `vibe-tether-deep` before product code is written.

Use Deep when the request explicitly asks for fact checking or confirmation before work, or when product behavior, UI direction, architecture, durable data, security, permissions, destructive operations, migration, deployment, publication, or release scope is unresolved. A filename, test target, or confident Agent inference does not count as user approval.

Start consequential work with `vibetether step start`; finish it with fresh command or artifact evidence. Never read raw runtime state, provider catalogs, unselected Skills, or unselected Experience. Never activate project direction or weaken security, data, permission, destructive-operation, merge, deployment, release, or publication gates without the required user decision.

Without a mandatory host hook, these controls depend on Agent cooperation; never describe an Agent-supplied confirmation flag as cryptographic proof of user identity.
